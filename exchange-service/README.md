# Exchange Service

`exchange-service` je Spring Boot mikroservis zaduzen za:

- dnevni fetch deviznih kurseva sa eksternog API-ja
- lokalno cuvanje kurseva u PostgreSQL bazi
- izlaganje kursne liste drugim servisima i klijentima
- kalkulaciju valutne ekvivalencije kroz `RSD` kao baznu valutu

Servis koristi:

- `Spring Boot 4`
- `PostgreSQL`
- `Liquibase`
- `Spring Security` sa JWT resource server pristupom
- `Spring Scheduler` za dnevni refresh kurseva

## Sta je implementirano

- `ExchangeRateEntity` i JPA repozitorijum za lokalni storage
- integracija sa Twelve Data providerom
- automatski dnevni cron fetch u `08:00`
- fallback na poslednji raspolozivi lokalni snapshot ako eksterni API padne
- endpointi za listu kurseva i pojedinacni kurs
- endpoint za kalkulaciju konverzije
- provizija kao konfigurabilan procenat
- unit i web testovi za servis, kontroler, scheduler i external client

## Podrzane valute

Podrzane valute su fiksirane u kodu:

- `RSD`
- `EUR`
- `CHF`
- `USD`
- `GBP`
- `JPY`
- `CAD`
- `AUD`

`RSD` je bazna valuta i za nju se ne radi eksterni fetch. Za `RSD` se koristi sinteticki kurs `1`.

## Biznis pravila

Osnovna pravila servisa:

- svi obracuni idu kroz `RSD`
- dnevni kursevi se cuvaju lokalno
- banka cuva i `buyingRate` i `sellingRate`
- za klijentsku kalkulaciju koristi se `buyingRate` izvorne ne-RSD valute kada se ulazi u `RSD`, a `sellingRate` ciljne ne-RSD valute kada se izlazi iz `RSD`
- ako eksterni fetch ne uspe, koristi se poslednji raspolozivi lokalni snapshot

Pravila kalkulacije:

- ista valuta: vraca se isti iznos
- `RSD -> strana valuta`: deli se sa `sellingRate` ciljne valute
- `strana valuta -> RSD`: mnozi se sa `buyingRate` izvorne valute
- `strana valuta -> strana valuta`: prvo `from -> RSD`, zatim `RSD -> to`

Primer:

```text
EUR -> USD
100 EUR * EUR.buyingRate = amountInRsd
amountInRsd / USD.sellingRate = finalUsdAmount
```

Bankarska marza:

```text
buyingRate  = marketRate * (1 - marginPercentage / 100)
sellingRate = marketRate * (1 + marginPercentage / 100)
```

Provizija:

```text
commission = fromAmount * (commissionPercentage / 100)
```

## Baza podataka

Schema se ne generise preko Hibernate-a. Hibernate je podesen na `validate`, a tabelu pravi Liquibase.

Glavna tabela:

```sql
CREATE TABLE exchange_rate (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    currency_code VARCHAR(3) NOT NULL,
    buying_rate NUMERIC(19, 8) NOT NULL,
    selling_rate NUMERIC(19, 8) NOT NULL,
    rate_date DATE NOT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT uk_exchange_rate_currency_date UNIQUE (currency_code, rate_date),
    CONSTRAINT chk_exchange_rate_buying_positive CHECK (buying_rate > 0),
    CONSTRAINT chk_exchange_rate_selling_positive CHECK (selling_rate > 0)
);
```

Lokacija Liquibase skripti:

- `src/main/resources/db/changelog`

Vazan princip:

- ne menjati vec izvrsene migracije
- za svaku izmenu seme praviti novi changelog fajl

## Struktura paketa

Najbitniji delovi servisa:

- `controller` REST endpointi
- `service` poslovna logika
- `client` poziv eksternog FX providera
- `domain` domen modeli, ukljucujuci `ExchangeRateEntity`
- `repository` JPA pristup bazi
- `dto` request/response modeli
- `advice` centralizovan exception handling
- `config` konfiguraciona svojstva i REST client setup

## API endpointi

Interno u samom servisu rute su:

- `GET /info`
- `POST /rates/fetch`
- `GET /rates`
- `GET /rates/{currencyCode}`
- `GET /calculate?fromCurrency=EUR&toCurrency=USD&amount=100`

Kroz API gateway iste rute su dostupne pod prefiksom:

- `GET /api/exchange/info`
- `POST /api/exchange/rates/fetch`
- `GET /api/exchange/rates`
- `GET /api/exchange/rates/{currencyCode}`
- `GET /api/exchange/calculate?fromCurrency=EUR&toCurrency=USD&amount=100`

Napomena:

- prefiks `/api/exchange` ne dodaje sam servis
- prefiks dodaje `api-gateway`

## Endpoint detalji

### `POST /rates/fetch`

Rucno pokrece fetch dnevnih kurseva za sve podrzane strane valute i cuva ih lokalno.

Koristi istu logiku kao scheduler.
Endpoint je zasticen i namenjen admin korisnicima.

Endpoint vraca i informaciju da li je koriscen fallback:

- `fallbackUsed`
- `sourceSnapshotDate`

### `GET /rates`

Vraca kursnu listu.

Opcioni query parametar:

- `date=YYYY-MM-DD`

Ako `date` nije prosledjen, vraca se poslednji lokalni snapshot.

### `GET /rates/{currencyCode}`

Vraca jedan kurs za zadatu valutu.

Opcioni query parametar:

- `date=YYYY-MM-DD`

Primer:

```http
GET /api/exchange/rates/EUR
GET /api/exchange/rates/USD?date=2026-03-23
```

### `GET /calculate`

Radi kalkulaciju ekvivalencije.

Obavezni query parametri:

- `fromCurrency`
- `toCurrency`
- `amount`

Opcioni query parametar:

- `date`

Primer:

```http
GET /api/exchange/calculate?fromCurrency=EUR&toCurrency=USD&amount=100
```

Primer odgovora:

```json
{
  "fromCurrency": "EUR",
  "toCurrency": "USD",
  "fromAmount": 100.00,
  "toAmount": 94.50,
  "rate": 0.94500000,
  "commission": 0.70,
  "date": "2026-03-20"
}
```

## Konfiguracija

Servis cita konfiguraciju iz:

- `src/main/resources/application.properties`
- `exchange-service/.env`

Za lokalni razvoj koristi se `exchange-service/.env.example` kao polazna tacka.

Minimalan lokalni `.env`:

```env
JWT_SECRET=local_exchange_dev_secret_at_least_32_chars
TWELVE_DATA_API_KEY=replace_with_provider_api_key
```

Ako zelis eksplicitno da definises i bazu:

```env
EXCHANGE_SERVER_PORT=8085
EXCHANGE_DB_HOST=localhost
EXCHANGE_DB_PORT=5437
EXCHANGE_DB_EX_PORT=5437
EXCHANGE_DB_NAME=exchange_db
EXCHANGE_DB_USER=postgres
EXCHANGE_DB_PASSWORD=postgres
JWT_SECRET=local_exchange_dev_secret_at_least_32_chars
TWELVE_DATA_API_KEY=replace_with_provider_api_key
```

`EXCHANGE_DB_EX_PORT` koristi `docker compose` za host mapiranje PostgreSQL porta. Sama aplikacija koristi `EXCHANGE_DB_PORT`.

Najbitniji property-ji:

| Property | Opis |
| --- | --- |
| `spring.datasource.url` | konekcija ka PostgreSQL bazi |
| `spring.jpa.hibernate.ddl-auto=validate` | Hibernate ne pravi tabelu, samo proverava mapiranje |
| `spring.liquibase.change-log` | lokacija Liquibase migracija |
| `exchange.rates.margin-percentage` | procenat marze za formiranje `buyingRate` i `sellingRate` |
| `exchange.rates.commission-percentage` | procenat provizije za kalkulaciju |
| `exchange.rates.fetch-cron` | scheduler cron izraz |
| `exchange.rates.fetch-on-startup` | da li se fetch radi pri startu aplikacije |

## Kako servis radi pri startu

Ako je:

```properties
exchange.rates.fetch-on-startup=true
```

onda servis pri startu:

1. podigne Spring context
2. poveze se na bazu
3. Liquibase proveri i po potrebi izvrsi migracije
4. startup hook pozove fetch dnevnih kurseva
5. kursevi se sacuvaju u `exchange_rate`

Ako provider vrati gresku:

- koristi se poslednji raspolozivi lokalni snapshot
- on se kopira za novi datum ako je potrebno

## Lokalni razvoj

Preporuceni razvojni tok je:

- PostgreSQL u Docker-u
- `exchange-service` pokrenut iz IntelliJ-a

To daje:

- debugger
- lokalne logove
- jednostavan pregled baze

### 1. Napravi lokalni `.env`

U `exchange-service` folderu napravi `.env` po uzoru na `.env.example`.

### 2. Podigni samo bazu

Iz `exchange-service` foldera:

```bash
docker compose up -d postgres
```

### 3. Pokreni servis iz IntelliJ-a

Pokreni `ExchangeServiceApplication`.

Podrazumevani local DB pristup je:

- host: `localhost`
- port: `5437`
- db: `exchange_db`

Zasto `5437`, a ne `5432`:

- unutar Docker mreze Postgres radi na `5432`
- na tvojoj masini je expose-ovan na `5437`
- to je normalno kada projekat ima vise baza ili vise Postgres instanci

### 4. Proveri da li je fetch odradio insert

Ako je `exchange.rates.fetch-on-startup=true`, po startu servisa bi trebalo da imas redove u tabeli `exchange_rate`.

## Provera baze iz terminala

Ako zelis da proveris stanje baze iz terminala:

Iz `exchange-service` foldera:

```bash

```

Ulazak u PostgreSQL shell:

```bash

```

Korisni upiti:

```sql
\dt
SELECT COUNT(*) FROM exchange_rate;
SELECT MAX(rate_date) FROM exchange_rate;
SELECT currency_code, buying_rate, selling_rate, rate_date, created_at
FROM exchange_rate
ORDER BY rate_date DESC, currency_code ASC;
```

```bash
docker compose exec postgres psql -U postgres -d exchange_db -c "SELECT COUNT(*) FROM exchange_rate;"
docker compose exec postgres psql -U postgres -d exchange_db -c "SELECT currency_code, buying_rate, selling_rate, rate_date FROM exchange_rate ORDER BY currency_code;"
```

## Ceste nedoumice

### Zasto IntelliJ prijavljuje `Cannot resolve table exchange_rate`?

Zato sto IDE nema aktivnu konekciju ka pravoj bazi ili schema jos nije kreirana.

Resenje:

1. podigni PostgreSQL
2. pokreni servis da Liquibase napravi tabelu
3. povezi IntelliJ Data Source na tu bazu
4. refresh schema

### Zasto mi treba `JWT_SECRET` ako samo lokalno testiram?

Zato sto servis ima aktivan JWT decoder bean i security konfiguraciju.

Ne treba ti pravi produkcioni secret, dovoljan je lokalni development secret dovoljne duzine.

### Zasto Docker ponekad ne radi i javlja `dockerDesktopLinuxEngine` gresku?

To obicno znaci da Docker Desktop nije pokrenut.

Prvo startuj Docker Desktop, pa tek onda:

```bash
docker compose up -d postgres
```

## Napomene

- `ExchangeRateEntity` se nalazi u `domain` paketu, u skladu sa konvencijom ostatka repoa
- scheduler i rucni fetch endpoint koriste istu servisnu metodu `fetchAndStoreDailyRates()`
- ako startup fetch ne uspe i nema nijednog lokalnog snapshot-a za fallback, servis ce prijaviti odgovarajucu gresku
- provider payload validacija je stroga, ukljucujuci `timestamp` i decimalna polja
